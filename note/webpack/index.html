# 模块化打包工具

之前提过前端模块化 (opens new window)的重要性。

但是注意，在原生项目中，CommonJS、AMD 都无法使用，ES6 语法需将脚本标签的 type 改为 module，即便如此，模块之间也是混杂难分。

模块化打包工具应运而生，它能够提供前端模块化开发，以及代码压缩混淆、处理浏览器端 JavaScript 的兼容性、性能优化等强大的功能。

常见的打包工具有：webpack、parcel、Rollup、browserify。

目前 Vue、React 等前端项目，基本上都是基于 webpack 进行工程化开发的。

故我们必须掌握 webpack。

# 初识 webpack

初始化：npm init -y // 快速生成 package.json 文件，跳过向导，所有选项默认

初始化工具：安装 webpack、webpack-cli // 均在 package.json 有体现

因为有兼容问题，先安装低版本 webpack@4.16.5 webpack-cli@3.3.11，后续的 css-loader、style-loader、less、less-loader 也要降低版本。

$npm i webpack@4.16.5 webpack-cli@3.3.11 -D
$npm i less@3.9.0 less-loader@5.0.0 -D
$npm i style-loader@1 css-loader@3 -D

初始化文件目录：src（源码文件）、dist（打包后的文件夹）

打包：webpack ./src/index.js ./dist/bundle.js // 左边是入口文件，右边是出口文件，需确保存在。

由于 webpack 是基于 node.js 开发的打包工具，因此支持使用 node.js 相关的语法和模块，故可以在文件中使用 CommonJS 和 ES6。

# 配置 webpack

每次都要命令打包，太麻烦，配置文件 webpack.config.js，webpack 在开始打包构建之前，先读取此文件，基于给定的配置，对项目进行打包，此时在控制台输入 webpack 进行打包。

npm 允许 package.json 文件使用 scripts 字段定义脚本，并使用 npm run xxx 运行。

配置打包脚本，简化指令，优点是终端的命令都是全局的，但是配置脚本后优先找到本地的 webpack，毕竟项目使用的 webpack 和本地的不一定相同。

// package.json
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1",
    "dev": "webpack"
  }
  
// webpack.config.js
const path = require("path");
module.exports = {
  entry: "./src/index.js", // 入口文件
  output: {
    path: path.resolve(__dirname, "dist"), // __dirname 指的是当前文件在文件管理器的绝对路径
    filename:'bundle.js' // 打包文件，项目中可以不存在
  }, 
};

// 控制台
$webpack
$npm run dev

# webpack 进阶

即可以在 webpack.config.js 文件中配置哪些选项，包括：入口（entry）、输出（outout）、模式（mode）、loader（加载器）、plugin（插件）。

# mode（模式）

mode:'development' | 'production'

  • development - 开发环境 不会对打包生成的文件进行代码压缩和性能优化 打包速度快 ，适合在开发阶段使用

  • production - 生产环境 会对打包生成的文件进行代码压缩和性能优化 打包速度很慢 ，仅适合在项目发布阶段使用

# loader（加载）

因为原生 webpack 只能够处理 .js 文件，若需要处理 css、png 等文件，需要配置 loader 加载器。

通过 npm 安装需要使用的 loader，然后在 webpack.config.js 文件中的 module 中配置。

作用是让 webpack 拥有了加载和解析非 JavaScript 文件的能力。

webpack 配置 loader (opens new window)

$npm i css-loader style-loader -D // css-loader 负责加载 css 文件，style-loader 负责编译 css 文件
$npm i less-loader@5.0.0 less@3.9 -D // less 可将 less 转为 css，less-loader 加载 less 文件
$npm i url-loader -D // 处理图片 url，若图片未超过限制，编译为 base64 字符串，若超过，则报错，需再配置 file-loader
$npm i file-loader -D // 处理文件 file
$npm i babel-loader balel-core babel-preset-es2015 -D // 将 ES6 文件转化为 ES5，babel-core 编译 ES6，babel-preset-es2015 就是 ES5
module.exports = {
    // 前面是打包入口和出口
    output: {
        publicPath:'./dist'  // 打包产出的文件放置位置。
    }
    module: {
        rules：[ // 配置使用规则
            {
                test: /\.css$/, // 匹配 .css 文件
                use: ['style-loader','css-loader'] // 多个 loader 的加载规则是从右到左，所以先加载后编译
            },
            {
                test: /\.less$/, // 匹配 .less 文件
                use: ['style-loader','css-loader','less-loader']
            },
            {
                test: /\.(jpg|png)$/, // 匹配图片文件
                use: [
                  {
                    loader: "url-loader", // options 配置选项，包括图片大小限制，解析后的图片命名
                    options: {
                      limit: 8192, // 图片大小限制，将限制内的图片编译为 base64
                      name: "img/[name].[hash:8].[ext]", // 超过大小限制的图片会被编译为文件，可能造成不同路径但同名的图片冲突，通过修改文件名避免。[] 表示变量，哈希截取八位，ext 表示文件后缀
                    },
                  },
                ],
            },
            {
                test: /\.js$/,
                exclude: /(node_modules|bowew_components)/ // exclude 限制只转化 src 文件夹下的 js 文件，将安装的局部 webpack 取消转化 ES5
                use: {
                loader:'babel-loader',
                options:{
                   presets:['es2015']
                           }
                }
            }
        ]
    }
}

Tips：npm 的命令修饰符 有 i、-g、--save、--save-dev、-S、-D。 i 下载，install 缩写 -g 全局安装 --save 项目运行时依赖，安装包写入 dependencies 中 --save-dev 开发时依赖，安装包写入 devDependencies 中 -S --save 缩写 -D --save-dev 缩写

# webpack 配置 Vue

安装版本需为 vue@2，配置 vue-loader，安装后在任意地方引入，并实例化。

被挂载的元素，可以获取并响应实例事件和数据。

// src/index.html
<div id="app">{{message}}</div>

// src/index.js
import Vue from "vue"
let app = new Vue({
  el: "#app",
  data: {
    message: "hello vue",
    age: 100,
  },
  methods: {
    handleClick() {
      this.message = 10;
      this.new = 1000;
    },
  },
});

此时使用 webpack 打包会报错，原因是 vue 在构建发布时，有两个版本，默认情况下 vue 配置为 runtime-only，此模式下不允许有 template。

需手动将 vue 改为 runtime-compiler 模式，即允许有 template，即使没有 templat，html 也会被当作 template。

在 webpack.config.js 中，与 mode 同级，加入 resolve 属性，将其指向 vue 发行使用的版本。

resolve:{
    alias:{
        'vue$':'vue/dist/vue.esm.js'// 匹配 node_modules 下的 vue-dist-vue.esm.js
    }
}

此外，也可以直接在在引入时引入该版本

import Vue from 'vue' // 改为如下
import Vue from '../node_modules/vue/dist/vue.esm.js'
# runtime-only 和 runtime-compiler 的区别

二者写法不同，渲染 Vue 的过程不同。

// runtime-compiler
new Vue({
    template:'<App/>',
    component:{App}
})

// runtime-only
new Vue({
    render:h=>h(App)
})

完整的 Vue 渲染页面的过程为：template —> ast(抽象树) -> render -> virtual dom -> UI 真实 DOM。

runtime-compiler 的模式下，就遵循这个过程，而 runtime-only 模式，Vue 渲染页面简化为：render -> virtual dom -> UI 真实 DOM。

故由此可得，runtime-only 省去了两个步骤，性能更好，但同时不支持 template 。

真实开发应选择 runtime-only 的模式。

# plugin

plugin 是插件的意思，作用是对 webpack 进行功能扩展，如打包优化，文件压缩等。

# loader 和 plugin 的区别

loader 主要用于转换某些类型的模块，它是一个转换器。 plugin 是插件，它是对 webpack 本身的扩展，是一个扩展器。

# 生成 html 文件

生成一个自动引入打包文件的 html 文件，避免修改打包文件（命名、路径）时 index.html 引入错误。

$npm i html-webpack-plugin@3.2.0 -D
// webpack.config.js
const HtmlWebpackPlugin = require('html-webpack-plugin')
module.exports = {
	plugins:[
	new HtmlWebpackPlugin()
	]
}

此时，重新打包后，dist 文件下自动生成 index.html 文件

<html>
  <head>
  </head>
  <body>
    <script type='text/javascript' src='dist/bundle.js'></script>
  </body>
</html>

注意，此时明明 index.html 是和打包文件同级的，但是脚本引入却显示 'dist/bundle.js'。

这是因为 publicPath 导致 script 多了一个 dist，需在 webpack.config.js 中删除 publicPath，改为在 url-loader/file-loader 中单独配置 publicPath。

并且，生成的 html 文件是空白的，在 new 插件的时候传入模板文件

new HtmlWebpackPlugin({
	    template:'index.html' // 与 webpack.config.js 同级的 index.html
		})

# webpack 搭建本地服务器

因为更新 js、css、vue 文件时，需要重新打包才能更新，降低开发效率。

若在运行脚本上添加 --watch，打包后修改文件，刷新浏览器页面可以实现更新。

通过开启本地服务器达到实时更新，可以监听代码是否改变并实时编译，在内存中更新用于测试。

本地服务器并没有映射到硬盘（也就是 bundle.js 没有改变），也没有进行打包，最终只需在开发完成后进行一次打包，就可以发布。

npm i webpack-dev-server@3.3 -D // 和 webpack-cli 同级，皆是 3.3 多版本
// webpack.config.js
module.exports = {
    devServer:{
	  contentBase:'./dist',// 指定为哪一个文件夹提供本地服务，默认是根文件夹，填写 ./dist
	  host: "localhost", // 服务器的IP地址，可以使用 IP 也可以使用 localhost
	  port:, // 端口，也就是页面端口
 	  inline:true, // 页面实时刷新
	  historyApiFallback：// 在 SPA 页面中，依赖 HTML5 的 history 模式
    }
}
// package.json
{
    "scripts":{
     "dev":"webpack-dev-server --open",   // --open 表示立刻在浏览器中打开页面
     "build":"webpack" // 打包指令
    }
}

# webpack.congfig.js 文件分离

webpack 中，有些配置是发布时所需，例如压缩丑化；

但是多数配置是开发时所需，例如加载器。

建议将 webpack 配置文件分离成：开发时的配置文件、运行时需要的配置文件，基础配置文件。

将 webpack.config.js 文件拓展为 build 文件夹，其中包含： base.config.js（基础配置文件）和 dev.config.js（开发配置文件）和 prod.config.js（发布配置文件）

在开发和发布文件中引入基础配置文件，当开发时调用开发文件打包，即使用 base + dev，发布时调用发布文件打包，即使用 base + prod。

基础配置如下：

// prod.config.js
const UglifyjsWebpackPlugin = require('uglifyjs-webpack-plugin')
modulre.exports = {
    plugins:[
        new UglifyjsWebpackPlugin(),
    ]
}

// dev.config.jd
module.exports = {
    devServer:{
        contentBase:'./dist',
        inline:true
    }
}

使用 webpack-merge 将两个配置文件合并，并修改打包指令：

$npm i webpack-merge@4.1.5 -D

// prod.config.js
const baseConfig = require('./base.config')
const webpackMerge =require('webpack-merge')
module.exports = webpackMerge.merge(baseConfig,{
	plugins:[
	new UglifyjsWebpackPlugin(),
	         ]
})

// dev.config.js
const baseConfig = require('./base.config')
const webpackMerge =require('webpack-merge')
module.exports = webpackMerge.merge(baseConfig,{
	devServer:{
	contentBase:'./dist',
                inline:true
	                }
})

// package.json
{
    "scripts":{
       "build": "webpack --config ./build/prod.config.js",
       "dev": "webpack-dev-server --open --config ./build/dev.config.js"
    }
}

打包的产出 dist 文件夹与配置文件同级，而当前配置文件在 build 文件夹下，故打包产出为 build/dist/，需修改打包的 output 路径。

path.resolve(__dirname, 'dist') // 原本的
path.resolve(__dirname, '../dist') // 现在的

# 项目配置

配置文件变为 vue.config.js 文件，configureWebpack 属性中进行编写。

configureWebpack: Object | Function<(config)=>{}>

configureWebpack 若为对象，会在打包时通过 webpack-merge 合并到最终 webpack 中。

若为函数，已解析好的配置 config 为参数，通过直接修改参数来自定义 webpack。

但更常使用，自定义化的配置 webpack 属性是 chainWebpack: Function<(config)=>{}>。

Vue CLI 内部的 webpack 配置是通过 webpack-chain 维护的，链式编程，常用于定义具名的 loader 和 plugin。

// configureWebpack 为对象
module.exports = {
    configureWebpack:{
        module:{
            rules:[{
                test: /\.css$/,
                use: ["style-loader", "css-loader"],
            }]
        }
    }
}

// configureWebpack 为函数
module.exports = {
    configureWebpack:(config)=>{
        config.module.rules.push({
                test: /\.css$/,
                use: ["style-loader", "css-loader"],
            })
    }
}

// chainWebpack
module.exports = {
    chainWebpack:(config) => {
        config
        .module
        .rule('css')
        .test(/\.css$/)
        .use(["style-loader", "css-loader"])
        .loader(["style-loader", "css-loader"])
        .end()
    }
}

# webpack 分离打包

项目中，将入口文件打包为 app.[hash:8].js，将所有 /node_modules 中的第三方包打包到 chunk-vendors.[hash:8].js 中。

故第三方包过多时，会出现 chunk-vendors.[hash:8].js 文件过大的问题。

第一个分片有 webpack 的路由懒加载

export function importFun(path) {
  // 注释用于打包文件的命名
  return import(
    /* webpackChunkName: "anaLv" */
    "../components/" + path
  );
}

SplitChunks 插件分离，可以使得打包后的模块无修改时，不会进行重复打包，在 optimization 属性中设置 splitChunks 属性的 cacheGroups 属性。 optimization.splitChunks.cacheGroups

module.export = {
    chainWebpack:(config)=>{
        conifg.optimization.splitChunks({
        // 配置公有属性
            cacheGroups:{
                echarts: {
                // 私有属性有 test、priority、reuseExistingChunk、enforce
                // 其余属性不设时采用公有属性
                        chunks: 'all', 
                        test: /[\\/]node_modules[\\/]echarts[\\/]/, // 模块的路径
                        name: 'echarts',
                        priority: 90 // 优先级
                }
            }
        })
    }
}
