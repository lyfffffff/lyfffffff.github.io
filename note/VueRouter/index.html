# 前言

本篇笔记基于 Vue Router 官网

Vue Router (opens new window)

路由响应式渲染组件，和页面渲染有密不可分的关系，故先讲页面渲染的发展，最开始页面渲染是由后端渲染的，而后慢慢发展为前端渲染，最后变成 SPA。

# 后端渲染

后端渲染即前端只负责展示，没有或很少 js 和后端交互，结构为 css + html + jsp 或 css + html + php。

后端处理 url 和页面之间的映射关系，url 发送到服务器，服务器会通过正则对该 url 进行匹配，并且给某 Controller 进行处理，Controller 处理后生成 html 页面或数据， 返回给浏览器。

故后端渲染网页在服务器渲染完成，再返回给浏览器，而不是在浏览器端进行渲染。

Tips jsp（java server page）= 前端 + java 在一个页面

# 前端渲染

随着需求复杂化，页面不仅仅是展示，功能要求变高，后端又要处理数据，又要渲染页面，比较困难。

彼时 ajax 兴起，开启前后端分离的开发模式，即后端不提供完整的 html 页面，而是提供一些 API 使得前端可以获取 到 JSON 数据，前端处理 JSON 数据并在 html 页面拼接，然后展示在浏览器上，这就是所谓的前端渲染。

# 前后端渲染的优缺点

# 前端渲染

# 优点
  • 前后端分离，前端关注 UI、后端关注 API
  • 节约服务器
  • 富交互

由于路由和框架的兴起，前端渲染有了更多优点：

  • 前端开发单页面富应用 SPA
  • 局部刷新
  • 懒加载

我们的主角：路由，终于上线了，但是在说路由前，还是继续说完前后端渲染。

# 缺点
  • 浏览器响应慢
  • 不利于 SEO

# 后端渲染

# 优点
  • 后端有完整 html 页面，有利于 SEO 搜索引擎。
  • 浏览器只需解析 html，性能好。
  • 后端生成静态化文件，即缓存片段，减少数据库查询时间。
# 缺点
  • 开发效率低，前后端高耦合。
  • 占用服务器资源。
  • 前端一般写一个静态 HTML 文件，然后后端再修改为模板，模板变，前后端都得变（css、html）。

# 前后端渲染应用场景

比如企业级网站，主要功能是展示而没有复杂的交互，并且需要良好的 SEO，则这时我们就需要使用后端渲染；

而类似后台管理页面，交互性比较强，不需要 SEO 的考虑，那么就可以使用前端渲染。

另外，具体使用何种渲染方法并不是绝对的，比如某些网站采用首屏后端渲染，即用户最开始打开的页面采用的是后端渲染，这样就保证了渲染速度，而其他的页面采用前端渲染，这样就完成了前后端分离。

# 初识 Router

前面总结了，前端渲染造就了前后端分离，前端渲染的发展产生了更多功能：单页面富应用 SPA、局部刷新、懒加载等。

这里大多都依靠 Vue 和 Vue-router 实现。

SPA（Single Page Application）表示单页面富应用，指整个应用程序只有一个唯一完整的 HTML 页面，而其它所谓的页面（url 变化），其实都是组件片段而已，每次都只进行局部刷新，并且可以在组件活跃时才加载该组件，称为懒加载。

多页面中，url 和服务器的映射是，不同的 url 返回不同的 html；而单页面，不同 url 表示不同的路径，返回不同路由对应的 html 片段。

通过 location.href 获取页面 url，完整 url 的组成部分为：

协议://主机:端口/路径?查询#锚点，例如 https://baidu.com/s?ie=UTF-8&wd=百度

SPA 只修改 url 的路径、查询和锚点，并不会改变主机和端口，故不改变服务器地址，也不使页面整体刷新。

# vue-router 配置

// router/index.js
import Vue from 'vue'
import VueRouter from 'vue-router'
import Home from '_c/Home.vue'
Vue.use(VueRouter)
const routes = [
{  path:'', // 默认渲染
   redirect: '/home', // 重定向
   component: Home
},
{
    path:'/home',
    name:'home',
    components:Home
}]
const router =new VueRouter({
    routes,
    mode:history // 修改路由模式，因为默认的 hash 模式出现 '#/'
})
export default router

// main.js
import router from './router'
new Vue({
    el:'#app',
    router,
    render:h=>h(App)
})

# 路由展示

通过 <router-view> 组件展示路由，当路由被激活时，<router-view> 将会被替换为路由所指向的组件，故有几个 <router-view> 就显示几个该组件。

<router-view> 有一个 name 属性，表示路由指向的组件名。

// 此路由指向多个组件，默认渲染 Radio 组件
{
   	path: '/radio',
   	name: 'Radio',
   	components: {
   		default: Radio,
   		radio1: Radio1,
   		radio2: Radio2
   	}
}
<router-link to="/radio">radio</router-link>
<router-view name="radio1"></router-view> // 渲染 Radio1 组件
<router-view name="radio2"></router-view> // 渲染 Radio2 组件

# 路由跳转

分为声明式和编程式，通过 <router-link to="#"> 或 this.$router。

window 也提供了对浏览器的会话历史的访问的对象 history。

# router-link 标签

<router-link> 默认渲染成 a 标签，to 属性代表路由路径

  • tag：改变标签，例如：tag="button"，将 a 标签渲染成按钮 button 标签
  • replace：不可以返回，没历史记录，直接加上
  • active-class：修改类名
    • 注意：当 router-link 被响应时，会出现两个类，router-link-exact-active 和 router-link-active，类似于 a:hover
    • active-class 替换 router-link-active 类，还可以通过路由修改响应类，在 options 中加入：linkActiveClass:'className'

# this.$router

this.$router 访问路由实例，调用其 push 或 replace 方法，可以实现页面跳转。

  • this.$router.[push/replace](String | Object<{path,name,query,params,hash}>)

push 和 replace 的不同的是，前者会在导航时向 history 添加新记录（即可以后退），后者不会。

# history

简单常用的页面前进后退：

  • history.back() 后退

  • history.forward() 前进

  • history.go(num) 往前进/后退 n 个页面，go(1) 等同于 forward()，go(-1) 等同于 back()

添加和修改历史记录中的条目，通常配合 window.onpopstate 使用。

  • history.pushState(state,title,url?) 当调用该方法添加历史条目时，state 会被赋值给 history 的 state 属性。 调用此方法后，任意前进后退都会触发 window.onpopstate 回调。
  • history.replaceState(state,title,url?) 替换当前 url 而不是新建。 因为是替换，故 pushState 回退会先回到调用前 url， replaceState 会直接回到url 的前一个 url。 例如当前是 'url1' ，pushState 到 'url2'，按回退，回到 'url1'，若 replaceState 到 'url2' ，按回退直接到 'url1' 的上一级，'url1' 已经不复存在。

# 动态路由

普通的路由是字符串，内部使用 / 分路径。

配置路由时，在路径前加 ':' 表示动态路由，即路径为一个变量，根据页面传来的参数显示不一样的 url 地址，但是指向同一个组件。

例如，我们可能有一个 User 组件，它应该对所有用户进行渲染，但用户 ID 不同。

同学 A 显示 'user/A'，同学 B 显示 'user/B'，可以使用 '/user/:userId' 指代这一类路由，同学 A 时 userId 为 A，同学 B 时 userId 为 B。

使用 router-link 中设置动态路由：

<router-link :to="'/user/'+userId">
<router-link to="'/user/lyf"
data(){
    return {
      userId:'luoyufeng'
    }
}
// 最终页面路径为 '/user/luoyufeng'

点击这两个 router-link，此时修改路径指向同一个组件，采用复用，而不是销毁重建，故不会重新调用 create 等周期函数。

页面动态修改 userId 时，路由不会变化，需重新触发，例如再次点击 router-link 或 this.$router.push/replace。

# 路由嵌套

当不完全符合路由时，也是无法匹配的，例如跳转地址是：'user/luoyufeng/18'，但是配置的路由路径为 'user/luoyufeng'，点击时无法指向该路由。

嵌套路由指的是，router-view 所指向的组件也包含 router-view，若想渲染，需在路由使用 children 属性配置子路由。

{
    path: "user/:id",
    component: testRouter,
    children: [
      {
        path: "18", // 匹配 "/user/:id/18"
        component: testCW,
      },
    ],
}

此时跳转地址为 'user/luoyufeng/18' 时，第一层 router-view 渲染为 testRouter，第二层 router-view 渲染为 testCW。

跳转地址为 'user/luoyufeng' 时，第一层 router-view 渲染为 testRouter，第二层 router-view 渲染为空。

跳转地址为 'user/luoyufeng/20' 时，每一层 router-view 都渲染为空。

若想跳转地址为 'user/luoyufeng' 时也渲染第二层 view-router，可以配置一个 path 为空的子路由。

{
    path: "user/:id",
    component: testRouter,
    children: [
      {
        path: "", // 匹配 "/user/:id"
        component: testCW,
      },
      {
        path: "18", // 匹配 "/user/:id/18"
        component: testCW,
      },
    ],
}

# 路由进阶

# 路由守卫

可以在路由跳转前进行查验和拦截，根据全页面跳转和组件跳转，分为全局路由守卫，和组件路由守卫。

常见使用场景：用户未登录时，修改路由需默认跳转到登录页面

  • router.beforeEach((to, from, next?Function)=>{}:false|router|other) 前置路由守卫，在路由跳转前生效，若参数回调为 false 跳转失败，返回一个路由字符串/对象，则跳转至该路由，返回其他，正常跳转。 to 表示将要去往路由 from 表示来自路由 next 表示放行函数，必须且只能调用一次，可以调用多次，但最终有效的只有最先执行的。

    • 不传参则表示路由跳转确认；
    • 传参数 Object/String 时，表示跳转路由的信息，覆盖 to 参数，重新进行路由跳转，也就是还会再次触发 beforeEach；
    • 传参数 Error 实例时，路由终止，且将 error 传给 router.onError；
    • 传参数 false 时表示，路由终止，且路由强制修改会跳转前。

  • router.afterEach((to, from)=>{}) 没有 next 参数，因为他在导航后，已经无法改变路由跳转。 常常用于路由跳转完后的初始化。

# next() 的坑

  • 调用多个只会执行一个
router.beforeEach((to, from, next) => {
  next("/test"); // 每次路由跳转只执行此
  next();
});
  • 路由跳转的回调地狱
router.beforeEach((to, from, next) => {
  next("/test"); 
});

本意是想让路由跳转到 '/test'，但是由于每次修改路由，又会再次调用路由守卫，最终变成回调地狱，即 next 触发路由守卫，路由守卫触发 next。

router.beforeEach((to, from, next) => {
  // next("/test") 触发 
  router.beforeEach(('/test', from, next) => {
  // next("/test"); 触发 
      router.beforeEach(('/test', from, next) => {
        // next("/test"); 触发 
        router.beforeEach(('/test', from, next) => {
        // next("/test"); 触发 
        ...code...
        });
    });
  });
});

最终也不跳转，只会报错 Maximum call stack size exceeded，即超过最大调用堆栈。

  • 必须要有 next() 作为出口 要么不使用 next(?)，要么必须存在 next() 作为回调出口。
beforeEach((to, from, next) => {
   if(to.path === '/home') {
   	next('/logon')
   } else {
    // 如果要去的地方不是 /home ， 就放行
   	next()
   }
}

# 其他路由守卫

  • 定义路由时守卫 beforeEnter 只在进入路由触发，动态路由时不触发，需显示调用 next()，否则不进入
  • 路由组件内守卫 beforeRouteEnter // 进入该组件时触发，需显示调用 next()，否则不进入 beforeRouteUpdate // 动态路由时触发 beforeRouteLeave // 离开该组件时触发 即和 methods 等属性一样定义在组件内部

# 路由懒加载

即需要使用时才加载组件，

// 将
// import UserDetails from './views/UserDetails'

const UserDetails = () => import('./views/UserDetails')

const UserProfileEdit = () =>
  import(/* webpackChunkName: "group-user" */ './UserProfileEdit.vue')
